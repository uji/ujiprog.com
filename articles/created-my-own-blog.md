こんにちは、ujiです。
ビッグウェーブ？に乗ってブログ投稿用の個人サイト作りました。

https://ujiprog.com

あんまり強い動機はなかったんですが、コーディングエージェントの発展で自前のものを開発・運用するハードルも下がったので、勉強がてら軽い気持ちで作りました。
途中で飽きるかなとも思ったんですが案外楽しめました。

## 構成

[syumai/workers](https://github.com/syumai/workers) で簡単なハンドラーを実装し、HTML や OG 画像等アセットは自動生成のコマンドを実装してます。
[Astro](https://astro.build/) を使って今のスタンダードっぽくやるつもりが気づけば Gopher スタックになってました。

### syumai/workers

[syumai](https://x.com/__syumai) さん作のライブラリで、Go で書いた HTTP サーバーを Cloudflare Workers 上で簡単に動かせます。

https://github.com/syumai/workers

Go 標準に準拠した形でサーバーを実装できるので、とても感触よかったです。
ほぼ Claude Code が実装書いたので自分は読むだけではありますが、スッと頭に入ってきます。

Go は実行ファイルがランタイム（プログラム実行時に必要な部品や環境）への依存が強く、サイズがそれなりに大きくなりますが、今の所 Cloudflare Workers の制限も問題にはなってないです。

### アセットの自動生成

Markdown パーサーの yuin/goldmark (CommonMark 完全準拠の狂気のライブラリ）と html/template や image などの標準/準標準パッケージを使って素朴に実装してます。

https://github.com/yuin/goldmark

Markdown の AST を HTML に変換する部分は自分が使いたい Markdown 構文のみをミニマムにサポートしてます。

- URL のみの段落をOGカードに展開
- YAML front matter (Markdown 上部に書くフィールド) を使ったロジック実装

といった独自のルールもサクッと作れます。

## Claude Code

自分は Claude の Pro プランを契約して、Claude Code をメインのコーディングエージェントとして利用してます。
他のエージェントツールの方がモデルが賢い・コスパが良い、といった話はありますが、Claude Code はLLMの効果を最大限向上するためのエコシステム構築が常に先を行ってる印象があり、爆速で刷新し続けられるエコシステムの進化を個人開発でも身近に感じたい、というのがモチベーションになってます。

30分から2時間ほどコードを書かせたら、レートリミットで数時間利用できなくなりますが、その間はコードを読んだり記事を書いたり、FigmaでOG画像のテンプレートを作ったり、本や漫画を読んだり、コーヒー淹れたりと他のことをしてました。
子供の頃の「ゲームは1日n時間まで」の制約を思い出します。

この制約、煩わしくなることは多いです。一方で、休憩の時間を作ってくれたり、よりエージェントの精度を上げてコスト抑えたい、ハックしたい欲求や創作意欲が湧いたりするので、趣味の範疇の利用であれば良い面もあるのかも、と思ったりしました。
例えば、Pro プランではウェブ上のクラウドインフラで動く Claude Code がプレビューで使えます。スマホアプリからも利用できるので、出先などでも開発が簡単に進められ、制約の中でより無駄なく Claude Code を使える機能として重宝します。
レートリミットを避けるために、こういう機能への感度が高くなる、というのはあるかもしれません。
思えば子供の頃も、ゲーム時間の制約は布団の中や公園など見られない場所でバレないように遊ぶハック（というか不正）をしてた気がします。縛られるとやりたくなる。

あとは OpenCode など他ツールを試す機会にもなったのもよかったです。

https://opencode.ai/

## デザイン

デザインは Anthropic 公式の `frontend-design` Skill を使って決めてもらいました。

https://github.com/anthropics/claude-code/blob/main/plugins/frontend-design/skills/frontend-design

自分が指定したのは色くらいですかね。アイコンの色味を使ってやってくれと依頼しました。
フォントや配置、アニメーションも良さげなものを提案してくれて助かりました。使い古されたありきたりなものは避けるようになってるっぽいですね。([DM Sans font](https://fonts.google.com/specimen/DM+Sans) をこの開発で知りました。良いですね)

Claude Code にデザインの検証をさせる際は `vercel-labs/agent-browser` を使いました。スクリーンショットで確認するところまで勝手にやってくれます。
Skill での利用前提なので、MCPよりもコンテキスト消費が小さく収まります。

https://github.com/vercel-labs/agent-browser

css ライブラリを特に指定しなかった結果、素朴に css で実装がなされていますが個人的には読みやすくて良いです。

OG 画像は Claude Code のデザインを真似て、Figma で作ってみました。
Gopher がカードを齧っているところがこだわりポイントです。

![OGP](https://ujiprog.com/created-my-own-blog-1.png)

## 今後

所謂フレームワークチックなライブラリに依存しない形になったので、柔軟なコードベースになりました。こういうスタックを選びやすくなるのはコーディングエージェントの恩恵な感じがします。
面白いギミックの実装や、技術検証などができる遊び場として使っていければと思います。

OG画像は記事ごとに生成してアップロードしてますが、将来的には Workers で動的に生成できたりすると良いなーという構想があります。
Workers の容量が厳しくなってきたら TinyGo に移行できたりしても面白そうです。(syumain/worksers は TinyGo サポートもあります。すごい)

Go コードですが、割と古めの書き方が残ってたりします。Go 1.26 に入る予定の `go fix` コマンドで整理させてみようと思います。

https://github.com/golang/go/issues/71859
